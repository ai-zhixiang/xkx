## 一、产品定位

### 1.1 一句话定位

**手机上的轻量级 AI Agent——享客虾生态的内容分发出口。**

WeClaw 不是独立的聊天 App，而是**智能内容推送器**：享客虾/嗨卡做好内容（AI 写歌、嗨卡、MV、视频）→ WeClaw 即时通知 → 一键分发到各平台（微信/抖音/快手/小红书）。

### 1.2 核心价值

| 问题 | WeClaw 方案 |
|------|------------|
| 享客虾做好的内容出不去微信生态 | 统一收件箱 → 自动适配各平台格式 |
| 用户做了嗨卡/歌/MV，不知道往哪发 | 智能分发引导，一键跳转粘贴 |
| 微信 Bot 只能聊天，不能主动推送 | 原生推送能力（通知栏/前台服务） |
| 多平台（微信/抖音/快手）格式不同 | 自动转换格式（链接/竖屏MP4/笔记文字） |

### 1.3 用户群

| 角色 | 场景 | 核心需求 |
|------|------|---------|
| **创作者**（嗨卡用户） | 做了首原创歌 → 想发到朋友圈/抖音 | 一键导出适配各平台的格式 |
| **Bot 用户**（微信 Bot 粉丝） | 和 Bot 聊完想做内容分发 | 内容制作后自动推送到手机 |
| **铭道（管理者）** | 监控 Bot、管理内容、看数据 | 统一管理后台推送 |

## 二、关键场景

### 场景 A：创作完成 → 即时推送

```
用户通过 Bot 完成 AI 写歌
  → 享客虾生成 MP3 + MV
  → 微侠令推送通知到 WeClaw
  → WeClaw 通知栏弹出「你的新歌已就绪」
  → 用户点开 → 预览 → 选择分发渠道
  → 微信（链接） / 抖音（9:16视频） / 快手 / 保存到相册
```

### 场景 B：Bot 消息收件箱

```
用户微信 Bot 的对话历史
  → 同步到 WeClaw
  → 在 App 内查看完整记录
  → 支持语音、图片、文件消息查看
  → 重要消息可「标记」和「转发」
```

### 场景 C：内容分发中心

```
WeClaw 首页
  ├─ 最近内容（歌曲/嗨卡/MV/视频）
  ├─ 一键分享（微信好友/朋友圈/抖音/快手）
  ├─ 保存到相册
  └─ 查看统计数据（播放量/打开量）
```

## 三、功能需求

## 四、技术方案

### 4.1 架构

```
微侠令（推送触发）
    │
    ▼
WeClaw App（Android）
    │
    ├─ 前台服务 + 通知栏
    │     └─ 接收推送 → 弹出通知 → 点击打开内容
    │
    ├─ 内容预览
    │     ├─ 音频播放器（MP3）
    │     ├─ 视频播放器（MP4）
    │     └─ 嗨卡卡片 WebView
    │
    ├─ 分享引擎
    │     ├─ Web Share API（微信/系统）
    │     ├─ 保存到相册（媒体文件）
    │     └─ 复制链接（文字分享）
    │
    └─ 后端 API（复用 weclawd-2）
          ├─ POST /api/push/register — 注册设备
          ├─ GET  /api/notifications — 获取推送列表
          ├─ POST /api/notifications/read — 标记已读
          └─ GET  /api/content/{id} — 获取内容详情
```

### 4.2 技术栈

| 层 | 技术 |
|----|------|
| Android | Kotlin + Jetpack Compose |
| 推送 | 前台服务 + 长轮询（暂不接第三方推送 SDK） |
| 网络 | OkHttp + Retrofit |
| 多媒体 | ExoPlayer（音频/视频） |
| 后端 API | weclawd-2（MD-1 :8005）+ 微侠令消息板 |

### 4.3 微侠令推送协议

```json
// 微侠令 → WeClaw 推送消息体
{
  "type": "new_content",
  "title": "你的 AI 新歌已就绪 🎵",
  "body": "《黄歇口》— 春申君列传主题曲",
  "content_id": "music_12345",
  "content_type": "music",        // music / card / mv / video
  "preview_url": "https://...",
  "file_url": "https://...",
  "actions": [
    {"label": "🎵 试听", "action": "preview"},
    {"label": "📤 分享", "action": "share"},
    {"label": "💾 保存", "action": "save"}
  ]
}
```

## 五、非功能需求

| 需求 | 要求 |
|------|------|
| 推送延迟 | 内容完成后 < 30s 推送到 App |
| 后台保活 | 前台服务 + 通知栏常驻（类音乐播放器模式） |
| 离线缓存 | 最近 50 条推送离线可查看 |
| 省电 | 长轮询间隔 30s，非活跃时延长到 120s |
| 包体积 | < 15MB（不含媒体缓存） |

## 七、开放问题

1. **Web 版 vs 原生 App**：MVP 阶段是否先用 PWA 验证分发场景，再投入原生开发？
2. **第三方推送 SDK**：是否接入极光/个推提高送达率，还是自建长轮询保活？
3. **iOS 版本**：是否同步开发 iOS，还是等 Android 验证后再做？
4. **渠道适配**：微信（链接/小程序）、抖音（视频）、快手（视频）、小红书（图文）— 第一个版本支持哪些？

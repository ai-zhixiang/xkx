## 1. WeClaw 是什么——演进史

WeClaw 经历了两轮完全不同的产品定义：

### Phase 1: 「手机上的 Hermes」（2026-05-29 ～ 2026-05-31）

**定位**: AI 原生通讯 App + 手机文件管家
**目标用户**: 微信文件传输被限制的用户
**核心场景**: 文件传输补微信短板 → 嗨卡做钩子 → 云空间套餐
**架构**: 深圳主站 FastAPI (:8010) + Android Kotlin 原生 App
**结果**: Android 编译不可用，WebView/JS Bridge 方案被用户否定为"意义不大"

### Phase 2: 「享客虾 Bot 连接器」（2026-06-01 至今）

**定位**: 多 Bot 微信聊天接入层，让多个微信 Bot 共享一个 Hermes AI 大脑
**目标用户**: 铭道（3 个 Bot）、张燕（2 个 Bot）、熊铭道
**核心场景**: 多 Bot 同时接入 → 用户消息通过 iLink → 连接器统一 → Hermes AI 回复
**架构**: MD-1 广州 FastAPI (:8001/:8005) + iLink 协议连接器
**结果**: 3 个 Bot 正常运行，扫码绑定、暗号系统、上下文管理均就绪

### 现状

两个项目并存，但互不相通：

| 项目 | 位置 | 端口 | 状态 |
|------|------|------|------|
| weclaw (Phase 1) | 深圳主站 | :8010 | systemd activating auto-restart |
| weclaw-1 (Phase 2) | MD-1 广州 | :8001 + :8005 | 3 个服务正常运行 |

## 4. 产品定位检讨

### 4.1 当前定位：弱化版 Hermes 网关

WeClaw 目前的实际定位是 **iLink Bot → Hermes 的连接器**，本质上是一个协议桥接层。它解决的问题是"同一个后端同时服务多个微信 Bot"。

**核心价值**: 让多个微信 Bot 共享同一个 Hermes AI 大脑，统一记忆、统一暗号、统一上下文。

**核心竞争力**:
- 扫码即用（比 Hermes 原生的 config 配置更友好）
- 多 Bot 统一管理（Hermes 原生只支持单 weixin 平台）
- 暗号系统（用户自定义触发词）

**问题**: 它只是一个 Bot 连接器，不是一个独立产品。"WeClaw" 这个名字（WeChat + Claw = 微信爪）对不上现在的定位——不再抓文件了，变成了 Bot 桥接器。

### 4.2 命名建议

建议将项目更名为 **享客虾 Bot Gateway**，去掉 WeClaw 这个名字。或者保留 WeClaw 但明确解释为"微侠"（微小的侠客，指 Bot 小助手）。

### 4.3 应该不应该

| 应该做 | 不应该做 |
|--------|---------|
| Bot 连接器 + 统一管理 | 独立的 Android App |
| 跨 Bot 消息路由 | 自己的用户注册体系 |
| iLink 协议桥接 Hermes | 直调 DeepSeek（已关闭）|
| 暗号 + 快捷命令系统 | 独立的文件存储 |
| 多实例负载均衡 | 替代 Hermes |

## 5. 架构检讨

### 5.1 当前架构评价

**优点**:
- 分层清晰：连接器 ↔ 网关 ↔ Hermes
- Bot 配置 DB 热加载，新增 Bot 无需重启
- 扫码绑定流程完整
- 优雅关闭 + 自愈看门狗

**问题**:

**① 单点风险**
- 整个 weclaw-1 跑在 MD-1 一台机器上
- MD-1 系统盘 40G（实际可用空间紧张），数据盘未充分使用
- MD-1 一旦网络/SSH 出问题，所有 Bot 瘫痪

**② 无状态管理不足**
- 连接器有 `bot_tokens` 等全局 Dict，进程重启即丢
- 应该把这些移到 DB 或 Redis

**③ 推送层协议错配**
- iLink 协议不支持主动推送（`ret=-1`），但架构中设计了完整的推送队列
- 正确的做法应该是换协议或换推送路径

**④ weclawd-2 重复**
- 用户 API 端口 :8005 的功能应该合并到 :8001 网关
- 两个 gunicorn 进程浪费资源，也增加复杂度

### 5.2 建议架构

```
iLink Bot ──→ 连接器 ──→ Bot 网关 (:8001)
                              │
                     ┌────────┼────────┐
                     │        │        │
                  暗号匹配  对话路由  推送队列
                     │        │        │
                    SQLite  Hermes(:8089) Redis(计划)
```

- 合并 weclawd-1 和 weclawd-2 为单一网关
- Bot 会话 token 持久化到 DB（已部分实现）
- 推送队列出口改为智享家服务号客服消息（已验证可用）

## 6. API 设计检讨

### 6.1 现有 API

**Bot 网关 (:8001)**
- `POST /api/bot/webhook` — Bot 消息入口
- `POST /api/bot/push` — 推送消息入队
- `POST /api/oauth/wechat-redirect` — OAuth 跳转
- `POST /api/oauth/wechat-callback` — OAuth 回调
- `POST /api/bind` — 扫码绑定

**用户 API (:8005，不稳定)**
- `POST /api/user/info` — 用户信息
- `POST /api/chat/send` — 聊天发送
- `POST /api/history` — 历史消息

### 6.2 问题

- 两套 API 混在两个端口，调用关系不清晰
- 没有 OpenAPI/Swagger 文档
- 错误码不统一
- `POST /api/bot/push` 返回 `success: true` 但实际推送失败（隐藏问题）

### 6.3 建议

- 合并到单一网关，前缀分组：`/api/bot/*`、`/api/user/*`、`/api/push/*`
- 添加 FastAPI 的 OpenAPI 文档开关
- 推送 API 返回实际投递状态（成功/失败），不要只报"已入队"

## 7. Android 客户端检讨

### 7.1 现状

- 完整的 Kotlin + Compose 项目（17 个 Kotlin 文件）
- 8 个 Compose 页面（登录/注册/聊天/设置/文件等）
- 后端指向 `hai.pangoozn.com/api/*`
- 编译问题已解决（`compileOptions` + 腾讯云 Gradle 镜像）
- 但 **没有人用**——用户 5 月 31 日说"APP 不可用"

### 7.2 问题

**根本问题**: Android App 被设计为 Phase 1 的产品（独立 AI 通讯 App），但产品方向已经转向 Phase 2（Bot 连接器）。App 面向的场景是：
- 用户主动打开 App → 注册登录 → 聊天
而现在的实际场景是：
- 用户在微信里找 Bot 聊天

**结论**: 当前的 Android App 和 Bot 连接器是两条产品线，前者已不再适合。

### 7.3 建议

- **如果继续 Android App**：应该重定位为 Bot 聊天增强客户端（扫码绑定后的辅助工具，离线消息查看、文件管理等）
- **如果不继续**：果断删掉 Android 目录，不要留着占空间和误导

## 8. 集成方案检讨

### 8.1 与 Hermes 的集成

当前方案：连接器 → Hermes `/v1/chat/completions` API

**问题**:
- 连接器通过 `x-session-id` 让 Hermes 管理会话，但没有利用 Hermes 的 tool 调用（Hermes 在连接器模式下被当作纯文本 AI）
- 暗号匹配在连接器和网关双重实现（代码冗余）
- 连接器启动时手动从 DB 加 Bot 配置，没有利用 Hermes 的自动发现

### 8.2 与 ailuckycards 的集成

**问题**:
- ailuckycards 的服务号和 weclawd 的 Bot 是两套独立的微信接入
- 用户身份不互通（channnel_bindings 表存 openid 但 ailuckycards 不读这个表）
- 推送能力不对等：ailuckycards（智享家）微信客服消息可用，weclawd Bot 推送不可用

### 8.3 建议集成方案

```
智享家服务号 ──→ ailuckycards ──→ 制卡/听歌/付费
                    │
                    │ (共享 openid)
                    ▼
iLink Bot     ──→ weclawd ──→ Hermes ──→ AI 聊天/暗号
                    │
                    │ (通过 openid 查 channel_bindings)
                    ▼
              推送消息到智享家服务号客服消息
```

关键：打通 `openid` 这一层。ailuckycards 和 weclawd 都存 openid，ai-rock 用户身份应该能跨系统识别。

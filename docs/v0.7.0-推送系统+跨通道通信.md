## 二、Session 上下文修复

### 问题
连接器用 `_call_hermes()` 调 Hermes `/v1/chat/completions` 时，手动从 `conversation_messages` 表拉10条历史拼接成 messages 数组。Hermes 不认 session，挂起/重连后上下文丢失。

### 修复
```python
# bot_gateway.py _call_hermes()
session_id = "xiakexia:" + session_key if session_key else ""
headers = {"Content-Type": "application/json"}
if session_id:
    headers["x-session-id"] = session_id

resp = await client.post(hermes_url, json={...}, headers=headers)
```

### 效果
- Hermes 按 `x-session-id` 管理会话生命周期
- 自动截断、摘要、过期
- 重连后上下文自动恢复
- DB 备份写入（`conversation_messages`）保留，双重保险

## 三、推送系统

### 架构

```
调用方（Hermes/其他模块）
    │ POST /api/bot/push
    ▼
weclawd-1 网关 (:8001)
    │ 写入 push_queue 表
    ▼
weclaw-connector _process_push_queue()
    │ 每5s轮询 pending 消息
    │ 查 user_context_tokens 表 → 拿 context_token
    │ 调 iLink sendMessage API
    ▼
用户微信收到消息
```

### 数据表

**push_queue**（weclawd 库）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | SERIAL PK | 自增主键 |
| bot_id | VARCHAR(128) | 目标 Bot |
| to_user | VARCHAR(256) | 目标用户 iLink ID |
| content | TEXT | 消息内容 |
| context_token | VARCHAR(256) | 会话令牌（可选） |
| status | VARCHAR(16) | pending / sent / failed |
| error | TEXT | 失败原因 |
| created_at | TIMESTAMP | 入队时间 |
| sent_at | TIMESTAMP | 投递时间 |

**user_context_tokens**（weclawd 库）
| 字段 | 类型 | 说明 |
|------|------|------|
| bot_id | VARCHAR(128) | Bot ID |
| to_user | VARCHAR(256) | 用户 iLink ID |
| context_token | VARCHAR(256) | 最新会话令牌 |
| updated_at | TIMESTAMP | 更新时间 |
| PRIMARY KEY | (bot_id, to_user) | |

### 推送 API

```
POST /api/bot/push
Content-Type: application/json

{
    "bot_id": "b8d69f48fa36@im.bot",
    "to_user": "o9cq806n88EiZCsWOatmsBO18miM@im.wechat",
    "content": "消息内容",
    "context_token": "",   // 可选
    "from_name": "智享家"    // 可选，自动格式化内容
}

响应: {"success": true, "message": "已入队，连接器将异步投递"}
```

**from_name 参数**（v0.7.1 新增）

- `from_name` 不为空时，`content` 自动格式化为 `来自《{from_name}》：{content}`
- 示例：`from_name="熊铭道", content="晚上一起吃饭"` → 实际推送 `来自《熊铭道》：晚上一起吃饭`
- 这样接收方不会莫名奇妙收到一条消息——知道是谁在说话

### context_token 自动保存

在 `unified_connector.py` 的消息处理循环中，每次收到用户的 `context_token` 后自动持久化到 `user_context_tokens` 表：

```python
if ctx:
    bot.context_tokens[from_user] = ctx
    # 持久化 context_token 到 DB（推送用）
    conn = await asyncpg.connect(DB_DSN)
    await conn.execute(
        "INSERT INTO user_context_tokens (bot_id, to_user, context_token) "
        "VALUES ($1, $2, $3) "
        "ON CONFLICT (bot_id, to_user) DO UPDATE "
        "SET context_token = $3, updated_at = NOW()",
        bot.bot_id, from_user, ctx
    )
```

**注意**：用户需要至少发一条消息后，`context_token` 才会保存到 DB，推送才能携带有效 token。

## 四、跨通道发消息

### 对话命令

在任何 iLink Bot 对话中，可以直接发：

```
发给{昵称} {消息}
```

例如：
```
发给熊铭道 晚上一起吃饭
发给铭道 测试通过了
```

### 处理流程

1. `bot_webhook_internal()` 先匹配 `^发给\s*(\S+)\s*[：:，,\s]+(.+)$`
2. 在 `channel_bindings` 表按 `nickname` 查找目标用户
3. **查发送者昵称**：从 `channel_bindings` 按 `channel_user_id` 查当前用户的昵称
4. **格式化内容**：消息自动加上 `来自《{发送者昵称}》：` 前缀
5. 写入 `push_queue` 表
6. 连接器轮询投递
7. 返回 `✅ 已发送给「{昵称}」`

### 查找优先级

1. `channel_bindings.nickname` — 已绑定微信的用户
2. `bot_accounts.nickname` — Bot 的显示昵称（未绑定的 Bot 本身）
3. 都查不到 → 返回 `❌ 找不到用户「{昵称}」`

## 七、v0.7.1 更新（2026-06-07）

### 新增：发送者身份标识

**推送 API 增加 `from_name` 参数**
- 文件：`routes/bot_gateway.py`
- 新增 `from_name` 请求参数
- `from_name` 不为空时，`content` 自动格式化为 `来自《{from_name}》：{content}`
- 避免接收方突然收到一条消息时莫名其妙

**`发给` 命令自动查发送者昵称**
- 文件：`routes/bot_gateway.py`
- 执行 `发给{昵称} {消息}` 时，自动从 `channel_bindings` 表查发送者的 `nickname`
- 查不到则默认显示"智享家"
- 内容格式化为 `来自《{发送者昵称}》：{消息}`

### 已知限制

**iLink Bot 主动推送不可用**
- 所有推送请求（`POST /api/bot/push`）均返回 `ret=-1`，即使携带有效 `context_token`
- 根因：iLink 协议只允许 Bot **回复**用户消息，不允许 Bot **主动发起**消息
- 这并非代码 bug，而是协议层天花板
- 如需主动推送能力，应走**智享家原生通道的客服消息 API**（已验证可用）

### 架构决策

**通道心跳（channel_heartbeat）正式废弃**
- 原主服务器 `channel_heartbeat.py` cron job 已删除
- weclawd 连接器是上位替代——它直接维护 WebSocket 连接，断连即知，不需要"看日志猜死活"
- 不再重建
